# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** This repo's own `.beads/` is NOT a launchable store: it holds only `issues.jsonl` plus a `metadata.json` pinned to bd 0.26.0 (`database: beads.db`), with no materialized SQLite or Dolt DB on disk. Under bd v1.0.5, `bd list --json --all --limit 0` fails reproducibly with `Error: no beads database found`, so abacus's auto-refresh errors out and it won't launch against this directory. This IS the source of the recurring "periodically shows error about running bd list" symptom — auto-refresh polls every 10s (`DefaultAutoRefreshSeconds=10`) so a failing backend re-errors every 10s. Materialize the store from the jsonl (`bd init` / import) before running abacus on this repo.
- **[gotcha]** Do NOT put literal `\n` sequences inside a single Bash command string (especially when wrapping in `lean-ctx -c '...'`) expecting newlines — they are passed through literally and corrupt the command (`cd /path\necho` becomes `cd: /pathnecho`). Use real newlines, chain with `;`/`&&` on one line, or split into separate Bash tool calls.
- **[convention]** `internal/ui` has multiple EXACT-count tests that break when you add a UI row/hint: `help_test.go` `TestGetHelpSections` asserts per-section row counts (e.g. `ActionsHas11Rows` → bump to 12 when adding a help row), and `footer_test.go` `TestFooterHintSlices/GlobalHintsCount` asserts `len(globalFooterHints)`. Adding a help row in `help.go` or a hint in `footer.go` requires updating the matching count assertion (and any `…IncludesXxx` sub-test) in the same change.
- **[gotcha]** This repo sets `git config core.hooksPath=.beads/hooks`, so the pre-commit hook that actually runs is `.beads/hooks/pre-commit`, NOT `.git/hooks/pre-commit`. The AGENTS.md `grep .git/hooks/pre-commit` check can mislead — run `git config core.hooksPath` first (or `grep caliber "$(git rev-parse --git-path hooks/pre-commit)"`) to find the active hook (both currently carry the caliber sync).
- **[gotcha]** A test that calls `lipgloss.SetColorProfile(...)` mutates GLOBAL lipgloss state and leaks into other tests in the package — a color-dependent assertion elsewhere then fails ONLY in the full `go test ./internal/ui` run, not when the test is run in isolation (seen: a new label test set `termenv.TrueColor` and `TestParentFieldRootModeInitialization/RootModeShowsNoParentIndicator` began failing under the suite but passed alone). Always save and restore: `prev := lipgloss.ColorProfile(); lipgloss.SetColorProfile(termenv.TrueColor); defer lipgloss.SetColorProfile(prev)`. (Note: the symbol is `termenv.TrueColor`, not `lipgloss.TrueColor`.)
- **[gotcha]** `.beads/issues.jsonl` is a PASSIVE export — `bd create`/`bd close`/`bd update` write only the Dolt DB, so a freshly created or closed bead is ABSENT from the jsonl (`grep -c <id> .beads/issues.jsonl` → 0). Run `bd export -o .beads/issues.jsonl` to regenerate it before staging/committing it alongside related code, or the committed jsonl silently lags the DB.
- **[gotcha]** lipgloss ends every rendered styled segment with a full SGR reset, which drops any background inherited from an enclosing style — so when composing multiple styled segments (label chips, separators, markers) inside an outer background-carrying column/panel style, only the FIRST segment paints on the intended bg and the rest fall to the terminal-default bg (observable as SGR `49m`, or a missing `48;2;` in a tmux `-e` capture, after segment 1). Fix: each segment AND separator must self-carry its background explicitly (`lipgloss.NewStyle().Background(bg).Render(...)`), and thread the row background through so highlighted/selected rows keep their selection bg rather than hardcoding the theme bg.
- **[fix]** After `cp`-ing a fresh build over the brew Cellar binary, running it may die with `Killed: 9` (exit 137). Cause: `cp` overwrites in place = SAME inode, so macOS keeps the OLD binary's CDHash cached for that vnode and the new content fails code-signature validation. Fix: re-sign ad-hoc after the copy — `chmod u+w TARGET && codesign --force --sign - TARGET && chmod 555 TARGET` — then `abacus -version` runs. (A new-inode atomic-swap via `mv` also sidesteps the cache.)
- **[gotcha]** The jcodemunch repo identity for this repo is `akimovmike/abacus`, NOT `local/abacus` — `register_edit`/search with `local/abacus` or with an absolute path errors `Repository index is not loadable` / `Path separator in name`. Confirm the id via `list_repos` first. `get_file_outline` also wants `file_path` (not `path`) plus the `repo` id — passing `path` alone errors `'repo' is a required property`, and passing `path`+`repo` errors `Provide exactly one of 'file_path' or 'file_paths'`.

- **[pattern]** TUI errors surface only as transient ~10s toasts and are otherwise discarded. To capture the full underlying error, launch `abacus --debug` and read `~/.abacus/debug.log` — but the log is TRUNCATED on every launch and only written when `--debug` (or config `debug: true`) is set.
- **[gotcha]** `make build` without `VERSION=` stamps the binary as `version dev`. Pass `VERSION=x.y.z` to embed a real version string.
- **[pattern]** To quantify a fix against `~/.abacus/debug.log`, capture before/after counts (`grep -c "bd run failed"`, `grep -c "context deadline exceeded"`). Because the log truncates per launch, each `abacus --debug` run is a clean trial.
- **[gotcha]** In `~/.abacus/debug.log`, a `bd run failed: ... ctxErr=context deadline exceeded: signal: killed` line means abacus's OWN context timeout killed the `bd` subprocess — NOT a genuine bd error. Check the `ctxErr=` field before blaming bd.
- **[gotcha]** `bd show`/`bd list` against a large real Dolt-backed store can block for many seconds when a concurrent external `bd` writer holds the embedded Dolt lock. Background loaders must give each call its own generous timeout, not one shared batch deadline, or a slow lock mass-kills every pending subprocess at once.
- **[gotcha]** Do NOT put literal `\n` sequences inside a single Bash command string (especially when wrapping in `lean-ctx -c '...'`) expecting newlines — they pass through literally and corrupt the command (a leading `cd /path` followed by literal `\n` becomes `cd: /pathnecho`; `sleep` + literal `\n` + `tmux` errors `sleep: invalid time interval`). Use real newlines, chain with `;`/`&&` on one line, or split into separate Bash calls.
- **[gotcha]** Serena's symbolic tools take DIFFERENT arg names and return a silent empty `{}` (no error) on no-match: `find_symbol` matches with `name_path_pattern` but returns `{}` for `name_path`, while `insert_after_symbol`/`replace_symbol_body` match with `name_path`. On an empty `{}`, the symbol usually exists — retry with the other parameter name before assuming it's missing.
- **[gotcha]** `tmux capture-pane`: use `-e` (lowercase) to KEEP ANSI escape sequences (needed to assert truecolor chip backgrounds, e.g. `48;2;224;108;117`); `-E` is the end-line flag and TAKES AN ARGUMENT, so `-peE` errors `capture-pane: -E expects an argument`.

- **[fix]** A multi-line `git commit -F - <<'EOF' … EOF` heredoc returned an empty result and left HEAD unchanged. Write the commit message to a file and use `git commit -F <file>` — more reliable in this harness and it sidesteps the literal-`\n` corruption trap.
- **[gotcha]** `git pull --rebase` aborts with `cannot pull with rebase: You have unstaged changes` because `bd` continuously rewrites `.beads/interactions.jsonl`. Don't `git add` it into a code commit — stash just that file around the pull: `git stash push -- .beads/interactions.jsonl` → `git pull --rebase` → `git push` → `git stash pop`, then `bd dolt push`.
- **[correction]** Label rendering for the tree labels column and the detail pane must NOT use `renderPillChip` — its nerd-font pill caps (`ue0b6`/`ue0b4`) render as tofu (□?) in terminals lacking a patched font. Use `renderLabelTag(label, hex)`: plain text on the theme background when no custom color is set, a padded colored block only when a custom color is configured. Verify by byte-counting the glyphs in a tmux capture.
- **[pattern]** To show a just-mutated per-issue field (e.g. a freshly added comment) immediately AND have it survive the next refresh, re-fetch it synchronously inside the command and apply it to the matching node (`applyCommentsToNode`) before `forceRefresh` — do NOT rely on the post-refresh background bulk loader, which is slow/contended under real embedded-Dolt load. `invalidateCommentCache` (background re-fetch) is only the fallback.
- **[gotcha]** `./scripts/tui-test.sh keys 'Enter'` (or `'Escape'`) sends the literal characters as runes, NOT the keypress — the overlay never applies/cancels and the feature looks broken. `keys` only works for printable runes (j/k/l/f/etc.); use the dedicated `enter` / `escape` subcommands for special keys.

- **[pattern]** A new persistent tree-view dimension (filter, view mode, toggle) must be threaded through FOUR places or it silently breaks: the `App` struct (`app.go`) AND the `viewState` struct (`state.go`), `captureState()` (`state.go`), and the restore block in `refresh.go` (next to `m.viewMode = state.viewMode`). Adding it to only `viewState` fails the build; skipping the `refresh.go` restore makes the value vanish on the ~10s auto-refresh. (Derived, index-based state like a multi-select anchor is instead reset in `recalcVisibleRows` since row indices shift on refresh/filter/view-mode.)

- **[gotcha]** `.golangci.yml` sets `run: tests: false`, so the `unused` linter does NOT count `_test.go` references — a production function called ONLY from tests is flagged `func (*App).X is unused` and `make check`/`make check-test` fails. This is the standard TDD trap: helpers written before they're wired into production code. Suppress with a `//nolint:unused // used in tests` directive placed on its own line directly above the func (or after the doc comment, separated by a blank `//` line), matching existing precedent in `layer_helpers.go`, `overlay_labels.go`, and `config.go`; run `gofmt -w` after adding it.
